# Ревью скриптов: simulate.py + calibrate.py
Дата: 2026-05-15

## Общая оценка

Оба скрипта хорошо структурированы и делают нетривиальную математику корректно.
Основные проблемы -- мертвый код и устаревшие комментарии после недавнего рефакторинга
`leontief_target`, один неиспользуемый импорт в каждом файле, и несколько констант
внутри цикла вместо модульного уровня.

## simulate.py — 908 строк, 8 функций + main

### Критические (исправлено)

**P1: Мертвый код `b_pseudoinv` (строки 189-197)**

Функция определена, но после замены `leontief_target` на прямое решение системы
B⁺ нигде в скрипте не вызывается. Dead code вводит в заблуждение (кажется, что
псевдообратная используется в симуляции).

```python
# было
def b_pseudoinv(B: np.ndarray) -> np.ndarray:
    """
    Псевдообратная матрица B⁺ с усечённым SVD ...
    """
    return np.linalg.pinv(B, rcond=0.01)

# стало — функция удалена
```

**P2: Неиспользуемый импорт `mticker` (строка 87)**

`matplotlib.ticker as mticker` импортируется, но ни одного вызова `mticker.*` нет.

```python
# было
import matplotlib.ticker as mticker

# стало — строка удалена
```

**P3: Устаревший комментарий в `step()` (строка ~322)**

Комментарий описывает старую B⁺-поправку, которая была ЗАМЕНЕНА на прямое
решение в `leontief_target`. После рефакторинга комментарий стал ложью.

```python
# было
# Ур-е (4.6)/(4.7): x*(t+1) = x(t) + B⁺(t) · [(I−A(t))·x(t) − u(t)]
x_star = leontief_target(x, A, B, u)

# стало
# (4.7): x*(t+1) = (I − A(t) − G_U·B(t))⁻¹ · u(t)
x_star = leontief_target(x, A, B, u)
```

### Значимые (исправлено)

**W1: `CLIP_B_FACTOR` и `CLIP_B_ABS` внутри тела `step()` (строки ~441-444)**

Две константы переопределяются на каждом из 30 вызовов `step()`. Должны быть
на уровне модуля рядом с другими параметрами.

```python
# было — внутри step()
CLIP_B_FACTOR = 5.0
CLIP_B_ABS    = 1.0
B_ceil = np.where(B0 > 0, CLIP_B_FACTOR * B0, CLIP_B_ABS)

# стало — модульные константы рядом с EPS_B, G_U
CLIP_B_FACTOR = 5.0   # B(t) не превышает 5× B(0) для ненулевых элементов
CLIP_B_ABS    = 1.0   # потолок для нулевых элементов B(0), млн евро / млн евро
```

**W2: Magic number `0.02` в `init_firms` (строка ~287)**

Коэффициент постоянных издержек зашит числом. Именованная константа документирует
смысл и позволяет менять одним числом.

```python
# было
c_jf = 0.02 * NU[j] * theta_mid * k_jf

# стало — модульная константа FC_RATE = 0.02
c_jf = FC_RATE * NU[j] * theta_mid * k_jf
```

**W3: Параметры `A, B` в `io_balance_residual` вводят в заблуждение**

На вызывающей стороне передаются `A_next, B_next` (значения t+1), но в сигнатуре
функции они называются `A, B`. Читающий тело функции не понимает, что это t+1.

```python
# было
def io_balance_residual(x, x_next, A, B, u): ...
    r = x_next - A @ x_next - B @ (x_next - x) - u

# стало
def io_balance_residual(x, x_next, A_next, B_next, u): ...
    r = x_next - A_next @ x_next - B_next @ (x_next - x) - u
```

### Мелкие

**N1: `List` из `typing` (строка 80) частично можно заменить**

`from __future__ import annotations` уже импортирован, поэтому в аннотациях
функций можно писать `list[int]`. Но на модульном уровне (`N_FIRMS: List[int]`)
аннотации не lazy-evaluated в Python 3.9 — `List` необходим. Всё корректно,
просто информация.

**N2: Docstring `b_pseudoinv` содержал описание rank-1 свойства B(0)**

Это описание теперь частично перенесено в комментарий к константе B0 (строка ~157).
Проверить что информация не потеряна. ✓ — уже есть: `# rank(B0)=1 → используется B⁺`.

## calibrate.py — 1164 строк, 12+ функций + main

### Критические (исправлено)

**P1: Неиспользуемый импорт `from scipy import linalg` (строка 19)**

`scipy.linalg` импортируется, но нигде не вызывается — весь линейный алгебры
идёт через `np.linalg`. Загрязняет пространство имён и создаёт зависимость.

```python
# было
from scipy import linalg

# стало — строка удалена
```

**P2: `import openpyxl` внутри функции `load_macro_params` (строка 362)**

Импорт внутри функции скрывает зависимость. Если пакет не установлен, ошибка
возникает только при вызове, не при старте скрипта. Все импорты должны быть
наверху.

```python
# было — внутри load_macro_params
import openpyxl
wb = openpyxl.load_workbook(...)

# стало — строка 19 (после numpy/pandas)
import openpyxl
# и в функции просто:
wb = openpyxl.load_workbook(...)
```

### Значимые (задокументировано, не изменено)

**W1: `calibrate_sector_params` мутирует входной словарь `macro` как побочный эффект**

```python
# внутри calibrate_sector_params
macro["GVA_series"][sid] = pd.Series(...)   # мутация аргумента
```

Функция должна возвращать данные, не записывать их в аргумент. Вызывающий код
не видит этого из сигнатуры. Рефакторинг затронет `main()` — отложено.

Рекомендуется: добавить `"gva_series": gva_series_dict` в возвращаемый dict,
убрать side-effect.

**W2: Декоративные разделители в `save_parameters`**

60+ строк `# ── Excel workbook: parameters.xlsx ──` и подобных. Семантически
эквивалентны двум пустым строкам, но загрязняют визуальный поиск кода.

### Мелкие

**N1: Лишняя аннотация типа в `seen: set = set()`**

В `parse_io_table` и `_parse_io_multi`:
```python
seen: set = set()   # тип очевиден из инициализатора
# → просто
seen = set()
```

**N2: `lines: List[str] = []` в `save_parameters`**

Аналогично — тип виден из `= []`, аннотация избыточна.
