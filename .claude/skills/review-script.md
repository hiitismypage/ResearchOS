Ты делаешь ревью Python-скрипта как старший разработчик. Твоя задача — найти реальные
проблемы и предложить конкретные исправления. Ты не переписываешь всё — ты указываешь
что и как исправить, с примерами кода.

## Аргумент скилла

Путь к скрипту или директории. Например:
- `/review-script scripts/simulate.py`
- `/review-script scripts/`

Если путь не указан — спроси пользователя.

## Что прочитать перед ревью

1. Указанный файл(ы) — полностью
2. Другие скрипты в той же директории — только заголовки и импорты (для понимания зависимостей)
3. Если скрипт читает данные — убедиться, что входные форматы совпадают с тем, что скрипт ожидает

## Чеклист ревью

### 1. Импорты и зависимости
- [ ] Все импорты используются?
- [ ] Нет циклических импортов или импортов внутри функций без причины?
- [ ] Порядок импортов: stdlib → third-party → local (PEP8)?
- [ ] Нет `import *`?
- [ ] Все необходимые зависимости импортированы (нет скрытых зависимостей от глобального состояния)?

### 2. Корректность логики
- [ ] Все граничные случаи обработаны (пустые массивы, None, деление на ноль)?
- [ ] Условия в if/else покрывают все ветки?
- [ ] Нет off-by-one ошибок в циклах и индексах?
- [ ] Типы данных согласованы (нет неявных преобразований, которые могут сломать логику)?
- [ ] Численные операции: нет переполнений, потерь точности, сравнений float через `==`?
- [ ] Генераторы случайных чисел: правильное зерно и воспроизводимость?

### 3. Переменные и именование
- [ ] Все переменные названы по смыслу, не однобуквенные (кроме математических обозначений)?
- [ ] Нет затенения переменных (переопределение встроенных: `list`, `type`, `id`)?
- [ ] Константы в UPPER_CASE?
- [ ] Нет "мусорных" переменных (`tmp`, `temp`, `data2`)?
- [ ] Согласованность стиля именования по всему файлу (snake_case vs camelCase)?

### 4. Функции и структура
- [ ] Каждая функция делает одно (single responsibility)?
- [ ] Функции не слишком длинные (>50 строк — сигнал к декомпозиции)?
- [ ] Нет повторяющегося кода, который можно вынести в функцию?
- [ ] Возвращаемые типы консистентны (не то dict, то list, то None)?
- [ ] Аргументы функций в разумном количестве (>5 — сигнал к dataclass или dict)?

### 5. Производительность
- [ ] Нет циклов Python там, где работают векторные операции NumPy?
- [ ] Нет повторных вычислений одного и того же внутри цикла?
- [ ] Матричные операции: `@` вместо `.dot()`, правильные размерности?
- [ ] Нет избыточных `.copy()` или повторных вызовов `np.array(...)`?
- [ ] Конкатенация списков: `list.append` + `np.array` в конце, не `np.vstack` в цикле?

### 6. Обработка ошибок и надёжность
- [ ] Проверки входных данных только на границах (не внутри вычислительных функций)?
- [ ] `try/except` ловит конкретные исключения, не голый `except`?
- [ ] Файловые операции закрыты через `with`?
- [ ] Нет `sys.exit()` внутри функций (только в `main`)?

### 7. Стиль и читаемость
- [ ] Строки не длиннее 100 символов?
- [ ] Нет магических чисел без именованных констант?
- [ ] Пустые строки между функциями (2 строки) и внутри функций (1 строка по смыслу)?
- [ ] f-строки там, где раньше был `.format()` или `%`?

### 8. Качество комментариев

Правила оценки комментариев как сеньор-разработчик:

**Хорошие комментарии** — только когда WHY неочевиден:
```python
# SVD-псевдообратная с порогом 1% от σ_max: подавляет числовой шум rank-1 матрицы
B_plus = np.linalg.pinv(B, rcond=0.01)

# Амортизационный фонд реинвестируется полностью — капитал не убывает при нулевой прибыли
k_new = (1.0 - delta) * k + I_tot
```

**Плохие комментарии** — убрать:
```python
# Вычисляем A (уже видно из кода)
A = Z / X

# Итерируем по секторам (очевидно)
for j in range(N):

# ─────────────────────── (декоративные разделители)
```

**Проверить:**
- [ ] Нет комментариев, которые повторяют то, что написано в коде?
- [ ] Нет "AI-style" декоративных паттернов (`# ───`, `# ===`, многострочные шапки)?
- [ ] Нет устаревших комментариев, описывающих удалённую логику?
- [ ] docstring только для публичного API; внутренние функции — без них или одна строка?
- [ ] Комментарии написаны как утверждения, не как вопросы или TODO-свалка?

## Формат вывода

```
# Ревью скрипта: <имя файла>
Дата: <дата>
Размер: <строки> строк, <функции> функций

## Общая оценка
[2-3 предложения: что хорошо, в чём главная проблема]

## Критические проблемы (исправить обязательно)

### P1: <название проблемы>
Расположение: `<функция>`, строка ~N
Проблема: <описание>
```python
# было
<проблемный код>

# стало
<исправленный код>
```

### P2: ...

## Значимые замечания (рекомендуется исправить)

### W1: <название>
...

## Мелкие замечания (по желанию)

### N1: ...

## Комментарии: что убрать / что добавить
[конкретный список с указанием строк]

## Итог
[что делать дальше: запустить тесты, профилировщик, etc.]
```

## Правила работы

1. **Читай весь файл** перед тем как давать замечания — не делай вывод по первым 50 строкам.
2. **Указывай строки** для каждой проблемы — не абстрактные "где-то в коде".
3. **Показывай было/стало** для каждого критического и значимого замечания.
4. **Не придумывай** замечания ради замечаний — если код корректен, скажи это явно.
5. **Не переписывай** весь скрипт — точечные правки.
6. Если скрипт математический (numpy/scipy) — проверяй размерности матриц явно.
7. Если скрипт делает I/O — проверяй, что пути к файлам не хардкодятся в теле функций.

Сохрани ревью в `<директория-скрипта>/review/<имя-скрипта>_review.md`.
Если директории `review/` нет — создай.
